# Nuggie Image Cards — Plan & Capability Reference

Status: **BUILT (Phases A–D + F)** — pending deploy verification (arm64 image build
+ run `/sync-rank-icons` once). · Owner: milestone-art initiative · Created 2026-06-21

Implemented: coin PNG pipeline (`@resvg/resvg-js` in the builder), reusable card
module `apps/bot/src/cards/` (`@napi-rs/canvas`, bundled fonts via fontsource
woff2), milestone announce now posts a rank-card embed with text fallback,
`/sync-rank-icons` admin command, and API payload enriched with
`lifetimeEarned`/`nextThreshold`/`nextLabel`. Card verified locally via
`apps/bot/scripts/preview-card.ts`. Bot typecheck clean.

Goal: give Nuggie (the Discord bot) a **reusable image-card rendering capability**,
and use it first for a **MEE6-style milestone rank-up card** that shows the tier
coin we built for the web. Designed so future cards (achievement unlock, weekly
Tide recap, casino big-win) reuse the same renderer.

This doc is both the build plan and the living reference for the capability.

## 2. Capability design — `renderCard()` (the reusable piece)

New module: `apps/bot/src/cards/` using **@napi-rs/canvas** (see §7 for why).

```
apps/bot/src/cards/
  canvasEnv.ts     # one-time font registration; exports { GlobalFonts ready }
  primitives.ts    # roundedRect, circleClip, fitText, drawAvatar, drawProgress,
                   # drawImageContain, palette helpers
  assets.ts        # cached fetch of remote PNGs (coin art, avatars) -> Image
  rankCard.ts      # renderRankCard(opts): Promise<Buffer>  (first consumer)
  index.ts         # re-exports
```

- `renderCard` primitives are generic; a "card type" is just a function that
  composes primitives and returns a **PNG `Buffer`** → wrapped in
  `AttachmentBuilder(buffer, { name: 'rank.png' })` and attached to an embed.
- `assets.ts` caches by URL (`Map<string, Image>`, TTL) so repeated posts don't
  re-fetch coins; avatars cached briefly (they change rarely).

### renderRankCard(opts) contract

```ts
type RankCardOpts = {
  displayName: string;
  avatarUrl: string;            // member.displayAvatarURL({ extension:'png', size:128 })
  tierLabel: string;            // "KAPPA"
  coinUrl: string;              // `${webOrigin}/art/milestones/<slug>.png`
  accent: string;               // tier color (hex) for ring/bar/glow
  bonus: number;                // ₦ paid
  lifetimeEarned?: number;      // for progress bar (see §8)
  nextLabel?: string;           // next tier name, if any
  nextThreshold?: number;       // for progress bar
  currentThreshold: number;
};
```

### Layout (≈ 900×280, MEE6-style)

```
┌──────────────────────────────────────────────────────────┐
│  ╭────╮   USERNAME                              ╭───────╮  │
│  │ AV │   reached                               │ COIN  │  │
│  ╰────╯   ★ KAPPA                               ╰───────╯  │
│           +₦75,000 bonus                                   │
│   ▓▓▓▓▓▓▓▓▓▓▓▓░░░░░░  250,000 / 750,000  → next tier       │
└──────────────────────────────────────────────────────────┘
```

- Background: dark base + subtle `accent` radial glow (no ripped art; original).
- Avatar: circular, `accent` ring.
- Coin: the tier PNG, right side, soft shadow.
- Progress bar: only if `lifetimeEarned`+`nextThreshold` present; else omit row.
- Apex tier (no next): show "APEX — undisputed" instead of a bar.

## 3. Coin asset pipeline (SVG for web + PNG for Discord)

Extend `apps/web/scripts/build-milestone-coins.mjs` with a rasterize pass using
**@resvg/resvg-js**: for each tier write `<slug>.svg` (today) **and** `<slug>.png`
(new, 512×512, transparent). The PNGs are committed to
`apps/web/public/art/milestones/` so they:

- deploy with the site and are URL-addressable (`/art/milestones/<slug>.png`),
- feed the rank card (`coinUrl`), role icons, and custom emojis — one source.

resvg renders our coins faithfully (it rasterizes the embedded base64 art + the
vector frame). No browser/puppeteer needed.

## 4. Wire the milestone rank-up card

In `processMilestoneAnnouncement` (`index.ts:806`):

1. Resolve `slug` + `accent` from `roleSettingKey` (ordinal `milestone_rank_NN`).
   Add a small ordinal→{slug,accent} table in the bot (mirrors
   `TIER_ROLE_KEYS_IN_LADDER_ORDER` already at `index.ts:723`). Slugs:
   `vault-dweller, silver, regular, divine, got-gud, king-of-the-hill, big-boss, kappa`.
2. `const png = await renderRankCard({...})`.
3. Post an `EmbedBuilder` with `.setImage('attachment://rank.png')` + the
   attachment, replacing the plain-text line (keep the `<@user>` mention + ₦bonus
   in `content` so the ping still fires).
4. On failure, **fall back to the current text line** (never block the role grant).

Role assignment / lower-tier cleanup stays unchanged.

## 7. Tech decisions

| Decision | Choice | Why |
|---|---|---|
| Card raster lib | **@napi-rs/canvas** | Prebuilt binaries incl. **linux-arm64-musl** (Graviton+alpine), no Cairo/apk deps; full text+image API. node-canvas would need apk cairo/pango/jpeg. |
| SVG→PNG (coins) | **@resvg/resvg-js** | Prebuilt arm64-musl; faithfully rasterizes our embedded-art SVGs; no headless browser. |
| Fonts | **Bundle TTFs** in `apps/bot/assets/fonts/`, register via `GlobalFonts.registerFromPath` | Alpine has **no system fonts**; must ship our own. Brand fonts (all OFL/Apache, license-clean): **Space Grotesk** (display), **Inter** (body), **JetBrains Mono** (₦ figures) — from `apps/web/src/theme.ts:194`. |
| Coin/avatar delivery | **Fetch by URL + in-memory cache** | Single source (web public). Avatars are URL-only anyway. No asset duplication into the bot image. |
| Card size | 900×280 @1x (optionally @2x=1800×560 for retina) | MEE6-like; Discord scales down cleanly. |

## 10. Deployment & risks

- **Arch match (highest risk).** Native prebuilds are arch+libc specific. The bot
  image must be built for **linux/arm64 + musl** to match the Graviton box. Verify
  `deploy.yml` builds the bot image for arm64 (buildx `--platform linux/arm64`, or
  build on the box). An x64-built image will crash at `import '@napi-rs/canvas'`.
- **Image size**: +canvas binary (~10–15MB) + fonts (~0.5MB). Acceptable.
- **Cold fetch**: first card fetches the coin PNG; cache thereafter. Avatar fetch
  per post (cached briefly).
- **Failure isolation**: card render/post wrapped in try/catch with text fallback;
  must never block the role grant or the bonus payout.
- **Single source of truth**: never hand-edit coin PNGs — regenerate via the build
  script so SVG (web) and PNG (Discord) stay identical.

## 11. Reusable capability reference (for future cards)

To add a new card type later (e.g. achievement unlock, Tide weekly recap):

1. Add `apps/bot/src/cards/<type>Card.ts` exporting `render<Type>Card(opts): Promise<Buffer>`.
2. Compose with `primitives.ts` (`roundedRect`, `circleClip`, `fitText`,
   `drawAvatar`, `drawProgress`, `drawImageContain`) — do not re-implement.
3. Reuse `assets.ts` for any remote images (cached). Reuse registered fonts from
   `canvasEnv.ts` (don't register fonts ad-hoc).
4. In the relevant `bot_announcements` handler, build the buffer + attach to an
   embed with `attachment://<name>.png`, always with a text fallback.
5. Keep layouts on the 900×280 grid for visual consistency unless a type needs
   its own format.

Conventions: PNG buffers (transparent where useful), `accent` color drives
per-context theming, all text uses bundled fonts, all remote art via `assets.ts`.
